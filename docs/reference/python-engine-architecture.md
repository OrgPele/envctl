## Invariants

- Supported CLI flags, compatibility commands, route selection, exit statuses, and user-facing command output must remain stable unless a task explicitly requires a compatibility-preserving update.
- `.envctl-state` artifact schemas, startup logs, debug reports, runtime readiness files, and state repository models are persistent contracts.
- Prompt installation behavior and plan-agent launch semantics are contracts across Codex, OpenCode, cmux, tmux, omx, ULW, headless, new-session, and direct-prompt paths.
- Generated contract artifacts are maintained by their generator scripts; update checked-in JSON only after comparing generated output.
- Generated worktree setup must stay confined to the current checkout or generated worktree targets. Do not mutate sibling worktrees as a side effect of planning changes.
- Serena is the symbol navigation tool for exact definitions, references, and diagnostics. CGC context `Envctl` is for broad repo graph analysis. Literal strings and docs/config text can use native search.

## How To Change Areas

- Runtime: add or move behavior into a focused `runtime/engine_runtime_*_support.py` owner, use `runtime/engine_runtime_construction.py` for object graph setup, `runtime/command_catalog.py` for supported command/flag registries, `runtime/lifecycle_blast_support.py` for blast-all ecosystem cleanup, `runtime/engine_runtime_planning_facade.py` and `runtime/engine_runtime_startup_facade.py` for compatibility wrapper groups, `runtime/engine_runtime_service_facade.py` for service/env/hook/command wrappers, `runtime/engine_runtime_action_facade.py` for action command wrappers, `runtime/engine_runtime_cli_facade.py` for CLI command wrappers, `runtime/engine_runtime_debug_facade.py` for debug command wrappers, and `runtime/engine_runtime_doctor_facade.py` for doctor/readiness/release-gate wrappers, then keep `PythonEngineRuntime` as a thin facade and run runtime dispatch/contract tests.
- Startup: extract lifecycle phases behind narrow inputs and result objects while preserving `StartupOrchestrator.execute` as the readable sequence; keep service env-file and backend env-contract behavior in `startup/service_env_support.py`, runtime prep cache/fingerprint behavior in `startup/service_runtime_state_support.py`, frontend bootstrap/dependency-install behavior in `startup/service_frontend_bootstrap_support.py`, and `service_bootstrap_domain.py` as the compatibility facade.
- Actions: route each action through action-owned helpers; keep `ActionCommandOrchestrator` as the compatibility entry point, use `action_command_test_facade.py` and `action_command_project_facade.py` for compatibility wrapper groups, keep project-action compatibility wrappers in `project_action_domain.py`, and split tests by action behavior. Review/analyze planning behavior belongs in `action_review_plan_support.py`; output rendering/stat pruning belongs in `action_review_output_support.py`; ship status orchestration belongs in `action_ship_support.py`; rich test-suite row spinner behavior belongs in `action_test_spinner_support.py`.
- State actions: keep route loading, filtering, health aggregation, and user-facing command selection in `state/action_orchestrator.py`; move reusable log/error payload behavior into `state/action_log_support.py` and preserve legacy private wrappers on the facade for hidden tests and CLI compatibility.
- Planning and worktrees: keep caller-facing helpers stable first, move cohesive internals into planning-owned modules, and preserve fresh AI worktree safety checks.
- Transports: share launch vocabulary through `planning/plan_agent/models.py` and helper modules; keep transport-specific process/session behavior in the transport modules.
- Requirements: split adapters behind stable adapter APIs. Shared adapter helpers belong in lifecycle model, policy, or cleanup owners rather than the container lifecycle facade. Supabase should be decomposed by configuration, lifecycle, readiness, database setup, auth users, repair, and reporting.
- Dashboard: keep orchestration, rendering, service row/status/configured/stopped-service rendering, AI session/run-command row resolution, dependency rendering/scope inference, PR-link root resolution/lookup/caching, backend resolution, interactive command dispatch/prompt/session helpers, failure details, stop/restart selectors, PR/review-tab flow, and target/service/test-scope selection separate; protect each owner with focused UI tests and keep `DashboardOrchestrator` wrappers for compatibility callers.
